    빠른 시작 (Quick Start)¶

    Port Framework로 .NET 장비 애플리케이션을 구축하는 단계별 가이드입니다.

    1. Document를 Page 단위로 변환하기 및 생성¶

    a. Page란 무엇인가?¶

    Page는 Port 시스템에서 데이터의 정의 단위입니다.
    하나의 .page 파일은 연관된 Entry(엔트리) 들의 집합으로 구성되며,
    Port 서버에 Push(등록)되면 메모리 내 데이터 저장소에 해당 항목들이 생성됩니다.

    Page = 하나의 기능 단위(예: IO, Sensor, Motor)에 속한 Entry 목록

    외부 문서(.docx, .xlsx, .csv)를 가져와 자동으로 Page를 생성하거나,
    C# 코드 내에서 [Page] 어트리뷰트를 통해 직접 정의할 수도 있습니다.

    b. Page의 구조와 역할¶

    .page 파일 구조¶

    [EntryKey]  [DataType]  [pkg:PackageName]  [property:{...}]
    필드 필수 설명
    EntryKey ✅ 엔트리의 고유 식별자
    DataType ✅ 데이터 타입 (f8, Enum.OnOff, char 등)
    pkg: 선택 연결할 Package 이름
    property: 선택 하드웨어 매핑 정보 (JSON)

    예시 (io.page):

    Bulb1OnOff    Enum.OnOff  property:{"IO.No":"D0.01","Model":"IODevice"}
Bulb2OnOff    Enum.OnOff  property:{"IO.No":"D0.02","Model":"IODevice"}
Bulb1Temp     f8          property:{"IO.No":"A0.01","Model":"IODevice"}
Bulb2Temp     f8          property:{"IO.No":"A0.02","Model":"IODevice"}

    Entry 구조¶

    Entry는 Port 시스템의 최소 데이터 단위입니다.
    각 Entry는 다음을 가집니다:

    • Key: 전체 경로 식별자 (카테고리.엔트리명, 예: Bulb1.OnOff)
    • DataType: 값의 타입 (수치형, 열거형, 문자형 등)
    • Value: 현재 저장된 값 (메모리 내 실시간 상태)
    • Property: 하드웨어 주소, 단위 등 부가 설정 (JSON)

    SECS 호환 데이터 타입:

    타입 설명
    f4 / f8 부동소수점 (4 / 8 바이트)
    i1 ~ i8 부호 있는 정수
    u1 ~ u8 부호 없는 정수
    char ASCII 문자열
    Enum.XXX 열거형 (사전 정의된 enum 참조)
    bool 불리언

    로컬 Entry 정의 (카테고리 전용)¶

    하드웨어 문서와 무관한 논리적 설정값(예: 목표 온도)은
    카테고리 폴더 내에 별도 .page 파일로 정의합니다:

    bulb1/.page:

    TargetTemp  f8

    c. Page를 Pull/Push 하는 방법¶

    Push — Entry를 서버에 등록¶

    방법 1: [Page] 어트리뷰트 클래스에서 Push

    // [Page] 클래스 인스턴스를 Push
Port.Push("sample", new CustomEFEM());

    방법 2: 문서(Document)에서 변환한 Page를 Push

    var ioDoc = Port.Document<IOModel>(@"C:\Users\admin\Documents\IO.docx");
Port.Push("sample", ioDoc.NewPage("Device"));

    방법 3: 디렉토리 전체를 Push

    // RepositoryInfo를 통해 폴더 전체를 REST API로 Push
Port.Push(repo);

    Pull — DB에서 파일 재구성¶

    Port.Pull은 CLI(port pull)를 호출하여 DB에 저장된 Entry 정의를
    .page, .enum 파일로 지정된 경로에 재생성합니다.

    // 문법
Port.Pull(string reponame, string root);

// 예시: D:\sample\Repo\pull\port\ 에 파일 생성
Port.Pull("sample", @"D:\sample\Repo\pull\");

    전체 초기화 패턴 (#if DEBUG)¶

    #if DEBUG
// 1. 프로젝트 루트 생성
Port.Repository.New(@"D:\sample\Repo\pull\", "sample");

// 2. 외부 문서에서 Page 생성
var ioDoc = Port.Document<IOModel>(@"C:\Users\admin\Documents\IO.docx");
ioDoc.Where(v => v.Key.Contains("OnOff")).ToList()
     .ForEach(v => v.DataType = "Enum.OnOff");
ioDoc.Where(v => v.Key.Contains("Temp")).ToList()
     .ForEach(v => v.DataType = "f8");

if (ioDoc.Count > 0)
{
    ioDoc.New(@"C:\...\io.page");
    ioDoc.New(@"C:\...\io.cs");
}

// 3. 인라인 Entry Push
Port.Push("sample", new CustomEFEM());

// 4. 문서 기반 Entry Push
Port.Push("sample", ioDoc.NewPage("Device"));

// 5. DB → 파일 재구성
Port.Pull("sample", @"D:\sample\Repo\pull\");
#endif

    2. Page를 Model과 맵핑 시키기¶

    a. Model이란 무엇인가?¶

    Model은 Port Entry들과 C# 프로퍼티를 연결하는 데이터 바인딩 계층입니다.

    • [Model] 어트리뷰트로 클래스를 선언합니다.
    • 각 프로퍼티는 [ModelBinding] 어트리뷰트로 특정 Entry에 연결됩니다.
    • Controller와 Flow에서 Model을 통해 Entry 값을 읽고 쓸 수 있습니다.

    b. Model과 Page의 관계¶

    .page 파일 (Entry 정의)
    ↓  Push
Port 메모리 DB (Entry 값 저장)
    ↕  ModelBinding
Model (C# 프로퍼티 ↔ Entry 매핑)
    ↕
Controller / Flow (비즈니스 로직)

    Page는 데이터 원천(Source of Truth) 이고,
    Model은 그 데이터를 코드에서 타입 안전하게 접근하는 뷰입니다.

    c. Model과 Page를 맵핑하는 방법¶

    [ModelBinding(instanceKey, entryKey)] 어트리뷰트를 사용합니다.

    • 첫 번째 인자: 컨트롤러 인스턴스 키 (예: "Bulb1", "LP1")
    • 두 번째 인자: Entry 상수 (자동 생성된 .cs 파일의 상수)
    [Model]
public class BulbModel
{
    // "Bulb1", "Bulb2" 두 인스턴스가 동일한 프로퍼티 구조 공유
    [ModelBinding("Bulb1", Io.Bulb1OnOff)]
    [ModelBinding("Bulb2", Io.Bulb2OnOff)]
    public Entry OnOff { get; set; }

    [ModelBinding("Bulb1", Io.Bulb1Temp)]
    [ModelBinding("Bulb2", Io.Bulb2Temp)]
    public Entry Temp { get; set; }

    [ModelBinding("Bulb1", Io.Bulb1TargetTemp)]
    [ModelBinding("Bulb2", Io.Bulb2TargetTemp)]
    public Entry TargetTemp { get; set; }
}

    Io.Bulb1OnOff와 같은 상수는 .page 파일에서 자동 생성된 io.cs에서 옵니다:

    // Auto-generated by Port. Do not edit manually.
namespace sample
{
    public static class Io
    {
        public const string Bulb1OnOff = "Bulb1OnOff";
        public const string Bulb1Temp  = "Bulb1Temp";
        // ...
    }
}

    3. Model과 Controller에 사용하는 방법¶

    a. Controller란 무엇인가?¶

    Controller는 하나 이상의 Flow를 포함하는 로직 컨테이너입니다.

    • [Controller] 어트리뷰트로 선언합니다.
    • Port.Add<TController, TModel>(instanceKey) 로 등록합니다.
    • 동일한 Controller를 여러 인스턴스(예: LP1, LP2)로 재사용할 수 있습니다.
    // Controller 등록
Port.Add<BulbController, BulbModel>("Bulb1");
Port.Add<BulbController, BulbModel>("Bulb2");
Port.Run();

    b. Flow란 무엇인가?¶

    Flow는 Controller 내부에 정의된 순차 실행 워크플로우입니다.

    • [Flow("FlowName")] 어트리뷰트로 내부 클래스를 선언합니다.
    • 각 단계는 [FlowStep(순서)] 메서드로 정의합니다.
    • 외부에서 Port.Set("Bulb1", FlowAction.Executing) 으로 실행을 트리거합니다.

    c. Flow에서 Model 핸들링 하는 방법¶

    Flow 메서드의 파라미터로 Model을 직접 받아 Entry 값에 접근합니다:

    [Controller]
public class BulbController
{
    [Flow("BulbOn")]
    public class BulbOn
    {
        [FlowHandler]
        public IFlowHandler handler { get; set; } = null!;

        // Step 0: 사전 조건 확인
        [FlowStep(0)]
        public void CheckInitialState(BulbModel model)
        {
            if (model.Temp.Value <= 100)
                handler?.Next();   // 다음 단계로 진행
        }

        // Step 1: 실행
        [FlowStep(1)]
        public void TurnOn(BulbModel model)
        {
            model.OnOff.Set("On");
            handler?.Next();
        }

        // Step 2: 완료 조건 모니터링
        [FlowStep(2)]
        public void MonitorTemperature(BulbModel model)
        {
            if (model.Temp.Value >= model.TargetTemp.Value)
            {
                model.OnOff.Set("Off");
                handler?.Next();   // Flow 완료
            }
        }
    }
}

    Flow 실행 / 중단:

    Port.Set("Bulb1", FlowAction.Executing);   // BulbOn Flow 시작
Port.Set("Bulb1", FlowAction.Canceled);    // 중단

    4. 다양한 Handler¶

    a. Handler의 종류¶

    Handler 인터페이스 용도
    IFlowHandler 기본 Flow 진행 제어 (Next())
    IFlowWithModelHandler<T> Model을 포함한 Flow 이벤트 구독
    ISchedulerHandler 이송(Transfer) 완료 스케줄링

    b. Handler의 역할과 사용 방법¶

    IFlowHandler — 기본 진행 제어¶

    [FlowHandler]
public IFlowHandler handler { get; set; } = null!;

handler.Next();   // 다음 FlowStep으로 이동
handler.Done();   // Flow를 즉시 Idle 상태로 종료 (동기)

    IFlowWithModelHandler\<T> — 이벤트 기반 Model 접근¶

    Flow 완료 / 단계 전환 / 오류 발생 시 Model을 함께 전달받습니다.
    [Preset] 메서드에서 구독하면 Flow 재실행 시에도 구독이 유지됩니다.

    [Controller]
internal class WTRController
{
    [Flow("Pick")]
    public class Pick : IFlowCACD<WTRCommModel>
    {
        [Handler]
        public IFlowWithModelHandler<WTRCommModel> handler { set; get; } = null!;

        [Handler]
        public ISchedulerHandler scheduler { set; get; } = null!;

        // Preset: 등록 시 1회 실행, 이벤트 구독은 영속
        [Preset]
        public void Preset()
        {
            handler.SetLogger(@"D:\log");
            handler.OnFlowFinished += (s, e) =>
                scheduler.TransferCompleted(e.Model.SelectedArm);
        }

        public void CheckStatus(WTRCommModel m) { Task.Delay(300).Wait(); handler.Next(); }
        public void Action(WTRCommModel m)      { handler.Next(); }
        public void CheckAction(WTRCommModel m) { handler.Next(); }
        public void Done(WTRCommModel m)        { handler.Done(); }
    }
}

    handler.Next() vs handler.Done()¶

    메서드 동작
    handler.Next() 상태 머신을 통해 다음 단계로 비동기 전환
    handler.Done() Flow를 동기적으로 즉시 Idle로 전환 후 OnFlowFinished 발생

    OnFlowFinished 콜백 내부에서 새 Flow를 시작해야 할 때는 반드시 handler.Done()을 사용하세요.
    handler.Next()를 쓰면 Flow가 아직 Idle 상태가 아니어서 AlreadyExecutingFlowException이 발생합니다.

    c. Handler 속성 테이블¶

    IFlowWithModelHandler\<T> 이벤트¶

    이벤트 발생 시점 전달 인자
    OnFlowFinished Flow가 Done 단계까지 완료 FlowFinishedWithModelArgs<T> — Model, 타이밍, 단계 기록
    OnFlowOccured 각 단계 전환 시 PortFlowOccuredWithModelArgs<T> — Model, 단계 상태
    OnFlowIssue 알람으로 Flow 중단 시 PortFlowIssueWithModelArgs<T> — Model, 알람 코드

    모든 이벤트 인자의 Model 프로퍼티는 해당 Flow 키에 바인딩된
    싱글턴 Model 인스턴스를 가리키므로, e.Model.Target.Value는 항상 현재 공유 메모리 값을 반영합니다.

    IFlowCACD\<T> — 표준 4단계 Flow¶

    장비 이송 Flow의 권장 인터페이스로, 메서드 순서가 고정됩니다:

    단계 메서드 설명
    0 CheckStatus 실행 전 사전 조건 확인
    1 Action 물리적 동작 실행
    2 CheckAction 동작 완료 여부 확인
    3 Done 완료 처리 — handler.Done() 호출로 Flow 종료

    5. Package 호출하기¶

    a. Package란 무엇인가?¶

    Package는 물리 장치(Bulb, Heater, IO 카드 등)와 Port Entry를 연결하는
    재사용 가능한 디바이스 드라이버 모듈입니다.

    • .page 파일에서 pkg:PackageName.PropertyName 구문으로 연결합니다.
    • [Package] 어트리뷰트로 C# 클래스를 선언합니다.
    • Entry 값이 변경되면 패키지의 해당 프로퍼티 setter가 자동 호출됩니다.

    b. Package API 호출 방법¶

    .page 파일에서 Package 연결¶

    Bulb1OnOff  Enum.OnOff  pkg:Bulb.OffOn  property:{"IO.No":"D0.01"}
Bulb1Temp   f8          pkg:Bulb.Temp   property:{"IO.No":"A0.01"}

    [Page] 클래스에서 Package 연결¶

    [PageEntry] 어트리뷰트의 Package 파라미터를 사용합니다:

    [Page("EFEM")]
public partial class CustomEFEM
{
    // pkg:Bulb1.OffOn 으로 Package에 연결
    [PageEntry(PortDataType.Enum, EnumName = "OffOn",
        Package  = "Bulb1.OffOn",
        Property = "{\"MIN\":0,\"MAX\":1}")]
    public const string LP1_BulbOnOff_o = "EFEM.LP1_BulbOnOff_o";
}

    Package 클래스 구현¶

    [Package]
public class Bulb
{
    [Logger]
    public ILogger Logger { get; set; }

    [Property]
    public IProperty Property { get; set; }

    [Valid("Device not connected")]
    public bool Valid() => serialPort.IsOpen;

    // Entry 값 변경 시 자동 호출되는 setter
    [API(EntryDataType.Enum)]
    public string OffOn
    {
        set
        {
            // 실제 하드웨어 제어 코드
            Logger.Write($"[INFO] Bulb OffOn → {value}");
        }
        get => _offOn;
    }

    private string _offOn = "Off";
    private SerialPort serialPort = new SerialPort();
}

    [Page] 클래스 Push¶

    // Port 서버 시작 전에 Push
Port.Push("sample", new CustomEFEM());

    6. Port.Set 기능에 대해¶

    Port.Set은 지정한 Entry의 값을 메모리 DB에 즉시 기록합니다.
    연결된 Package가 있으면 setter가 호출되어 실제 하드웨어 동작으로 이어집니다.

    기본 사용법¶

    // Entry 값 설정
Port.Set("Bulb1.OnOff", "On");
Port.Set("Bulb1.TargetTemp", "85.0");

// Flow 실행 / 중단
Port.Set("Bulb1", FlowAction.Executing);
Port.Set("Bulb1", FlowAction.Canceled);

    Model에서 Entry 직접 Set¶

    // Flow 내부에서 model 파라미터로 직접 Set
model.OnOff.Set("On");
model.TargetTemp.Set("80.0");

    IFlowModel에서 Set (@ 바인딩)¶

    [FlowModel]
public IFlowModel model { get; set; }

model.Set("@OnOff", "On");   // @ 접두사 = ModelBinding 키 참조

    7. Port.Get 기능에 대해¶

    Port.Get은 지정한 Entry의 현재 값을 메모리 DB에서 읽습니다.
    항상 최신 값을 반환하며, 고성능 인메모리 접근을 지원합니다.

    기본 사용법¶

    // Entry 값 읽기
string onOff = Port.Get("Bulb1.OnOff");   // → "On" or "Off"

if (Port.Get("Bulb1.OnOff") == "On")
{
    Port.Set("Bulb1.OnOff", "Off");
}

    Model에서 Entry 직접 Get¶

    // Entry.Value 프로퍼티로 현재 값 접근
double temp   = (double)model.Temp.Value;
string status = (string)model.OnOff.Value;

    IFlowModel에서 Get (@ 바인딩)¶

    var value = model.Get("@Temp");   // ModelBinding 된 Entry 값 조회

    8. Rule Script를 통해 Port Set/Get 동작 제어하기¶

    Rule Script는 Entry 값의 조건을 감시하여 자동으로 다른 Entry를 제어하는
    이벤트 기반 자동화 규칙 엔진입니다. .rule 파일에 정의하거나 C# 코드에서 직접 등록합니다.

    Rule 문법¶

    set("조건 Entry == 값", "실행 Entry >= 임계값")
    구성 요소 설명
    첫 번째 인자 트리거 조건 (Entry 경로 연산자 값)
    두 번째 인자 실행 조건 또는 Set 동작

    지원 연산자: ==, !=, >=, <=, >, <

    .rule 파일 예시¶

    // Bulb1.OnOff 가 "On"이 되고, Bulb1.Temp 가 80 이상이면 자동으로 Off
set("Bulb1.OnOff == On", "Bulb1.Temp >= 80")

    C#에서 Rule 등록¶

    // Port.Set을 통해 Rule 조건을 수동으로 트리거
Port.Set("Bulb1.OnOff", "On");

// Flow 내에서 조건 확인 후 Set
[FlowStep(2)]
public void MonitorTemperature(BulbModel model)
{
    if (model.Temp.Value >= model.TargetTemp.Value)
    {
        model.OnOff.Set("Off");
        handler?.Next();
    }
}

    전체 흐름 요약¶

    Entry 값 변경 (Port.Set)
    ↓
Rule Engine 조건 평가
    ↓ 조건 일치
자동 Port.Set 실행 → 연결된 Package setter 호출 → 하드웨어 동작
    ↓
Flow Step 전환 조건 재평가

    Rule Script는 Flow와 독립적으로 동작하며, 여러 Entry 간의 복잡한 연동 조건을
    코드 수정 없이 .rule 파일만 편집하여 변경할 수 있습니다.
